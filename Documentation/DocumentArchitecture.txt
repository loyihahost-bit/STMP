# STMP Documentation Architecture Initialization

You are working on the **Smart Traffic Management Platform (STMP)**.

This project follows a **Documentation-First Development** methodology.

Documentation is the single source of truth.

No implementation should begin before documentation is approved.

---

# Project Philosophy

STMP is an enterprise-grade intelligent traffic management platform designed for long-term scalability.

The documentation must be written as if this project will eventually be maintained by multiple software engineers, AI assistants, researchers, and government organizations.

The documentation should remain understandable for years.

Every document must be:

- professional
- structured
- modular
- scalable
- implementation-independent
- easy for both humans and AI assistants to understand

---

# Documentation Principles

Follow these principles everywhere.

1. Documentation First
2. Single Source of Truth
3. Modular Documentation
4. No Duplicate Information
5. Cross Reference Related Documents
6. Enterprise Documentation Style
7. AI-Friendly Structure
8. Human Readability
9. Scalability
10. Maintainability

---

Scope

Background

Definitions

Requirements

Workflow

Not every document requires every section, but the structure should remain consistent.

---

# Writing Style

Always use:

Professional English

Markdown

Short paragraphs

Clear headings

Consistent terminology

No duplicated information

No implementation details unless the document specifically requires them.

---

# Cross Referencing

Related documents must reference each other.

Avoid duplicated explanations.

One concept should be explained in one location only.

---

# Diagrams

Whenever architecture becomes difficult to understand through text, propose creating diagrams.

Prefer:

Mermaid

PlantUML

draw.io

Architecture diagrams should always have corresponding markdown documentation.

---

# Architecture Decision Records

Every important decision must produce an ADR.

Each ADR should include:

Context

Problem

Decision

Alternatives

Consequences

Status

References

ADR files are historical records.

Do not rewrite history.

---

# AI Assistant Rules

Never invent undocumented functionality.

Never change project architecture without documentation.

Never create undocumented APIs.

Never rename folders without updating documentation.

Always keep documentation synchronized.

Always preserve consistency across documents.

Documentation changes should happen before implementation changes.

---

# Long-Term Goal

This documentation should eventually become a complete engineering knowledge base for the Smart Traffic Management Platform.

A new developer or AI assistant should be able to understand the entire project only by reading the documentation.

The documentation should remain scalable, maintainable, and enterprise-grade for many years.

# Change Management

The AI assistant must not create, rename, move, or delete documentation files or folders without explaining the reason.

Before making structural changes, the assistant should evaluate whether the change improves maintainability, scalability, or clarity.

Backward compatibility of documentation should be preserved whenever possible.
